# Pre-1.0 Audit Remediation — Design Spec

**Date:** 2026-04-22
**Status:** Approved
**Approach:** Foundation First (refactor, then fix, then harden)

## Context

A comprehensive pre-1.0 audit identified ~35 issues across security, Rust core, Tauri commands, frontend, and configuration. Issues span 7 critical, 10 high, and 18 medium findings. Sync/relay is shipping in 1.0 and the relay server is operational, so all sync-related security issues are in scope.

No deadline pressure — quality gates take priority over speed.

## Remediation Strategy

8 sequential batches, each becoming a PR. Ordered so that earlier batches simplify later ones (constructor refactor unblocks data integrity fixes; input validation precedes sync hardening).

H5 (DefaultHasher instability in device.rs) is downgraded to document-only. The current device ID derivation is battle-tested across multi-device sync and must not be changed without extensive cross-device testing.

## Batch 1: Workspace Constructor Refactor

**GitHub label:** `Workspace Constructor Refactor`
**Issues addressed:** M3

### Problem

Four near-identical constructors (`create`, `create_with_id`, `create_empty`, `create_empty_with_id`) in `workspace/mod.rs` repeat ~150 lines of initialization code each (~600 lines total). Behavioral divergence is already creeping in — `create_empty_with_id` skips starter-script seeding via a subtle `let` vs `let mut` difference.

### Design

- Extract a `WorkspaceConfig` struct capturing the variable parts:
  - `workspace_id: Option<String>` — generate if None, use provided if Some
  - `insert_root_note: bool`
  - `seed_starter_scripts: bool`
- One private `fn init_core(config: WorkspaceConfig, ...) -> Result<Self>` that handles all shared initialization: device ID construction, workspace root derivation, workspace_meta insertion, attachment key derivation, script loading, identity pubkey computation, owner_pubkey storage, undo_limit setup, HLC init, permission gate setup
- Four public constructors become thin wrappers (5-10 lines each) that build a config and delegate
- Document the intentional behavioral difference (skip starter scripts) via the `seed_starter_scripts` field

### What doesn't change

Public API signatures remain identical. All existing tests pass without modification. No database schema changes.

### Testing

- Existing `#[cfg(test)]` workspace tests cover all 4 constructors
- Add one test verifying `create_empty_with_id` correctly skips starter scripts

## Batch 2: Data Integrity Fixes

**GitHub label:** `Data Integrity Fixes`
**Issues addressed:** C5, H6, H7, M4, M5, M6, M7

### C5 — Split transaction in `delete_note_recursive`

**File:** `workspace/notes.rs:1317-1340`

The note deletion and operation log entry are committed in separate transactions. Crash between them = notes gone but no DeleteNote op in the log, breaking undo and sync.

**Fix:** Advance HLC and clone signing key before opening the transaction. Run both the `DELETE FROM notes` and the `log_op(DeleteNote)` inside the same transaction. Follow the pattern used by every other mutating method.

### H6 — Migrations not in a transaction

**File:** `storage.rs:101-458`

~12 migration statements run as individual `conn.execute()` calls. Crash mid-migration leaves a partially migrated database.

**Fix:** Wrap `run_migrations` in `BEGIN IMMEDIATE` / `COMMIT`. SQLite DDL is transactional.

### H7 — `wall_clock_ms()` panics on pre-epoch clock

**File:** `hlc.rs:113-118`

`SystemTime::now().duration_since(UNIX_EPOCH).unwrap()` panics if the system clock is set before 1970.

**Fix:** Change `.unwrap()` to `.unwrap_or_default()`.

### M4 — ScriptRestore hardcodes category as "library"

**File:** `workspace/undo.rs:538-549`

Undoing a script delete restores it with `category = "library"` regardless of the original category. A schema-category script comes back as library, breaking schema loading.

**Fix:** Add `category: String` to the `ScriptRestore` variant. Capture the original category when the undo entry is created.

### M6 — `set_note_checked` stores HLC milliseconds in `modified_at`

**File:** `workspace/notes.rs:892-893`

Uses `ts.wall_ms as i64` (13-digit milliseconds) for `modified_at`, while every other write path uses `chrono::Utc::now().timestamp()` (10-digit seconds). Notes modified via checkbox appear to be from year ~57,000 AD.

**Fix:** Replace `ts.wall_ms as i64` with `chrono::Utc::now().timestamp()`. The HLC is still advanced for the Operation — just not used for the display timestamp.

### M5 — Op purge strategy too aggressive for sync

**File:** `workspace/mod.rs` (all constructors)

All constructors hardcode `PurgeStrategy::LocalOnly { keep_last: 100 }`. With sync enabled and active editing, 100 ops is exhausted quickly — a peer offline for a while misses operations and needs a full snapshot resync instead of a delta. The `undo_limit` (default 50) plus the 100-op purge means old undo entries can reference already-purged operations.

**Fix:** After the Batch 1 refactor, the purge strategy is set in one place (`init_core`). Make it configurable: increase the default to 1000 for sync-enabled workspaces, or make it a workspace_meta setting. The exact number is a tuning decision — 1000 is a reasonable starting point that covers a few days of active editing.

### M7 — `import_workspace` sets importer as owner

**File:** `export.rs:517-551`

When importing a workspace, the importer's identity becomes the owner (via `AllowAllGate`), regardless of who originally created the workspace. This is incorrect for shared workspace templates.

**Fix:** Preserve the original `owner_pubkey` from the imported workspace's `workspace_meta` rather than overwriting it with the importer's key. If the importer is not the original owner, they should receive the role that was assigned to them (or a default guest role), not owner.

### H5 — DefaultHasher instability (document only)

**File:** `device.rs:10-30`

Add a code comment explaining that `DefaultHasher` is technically unstable across Rust versions, but the current device ID derivation is battle-tested for multi-device sync and must not be changed without extensive cross-device testing. No code change.

### Testing

- C5: Test that delete + op log are atomic (verify op exists after delete)
- H6: Test migration rollback on simulated mid-migration failure
- H7: No direct test (system clock manipulation), but the `.unwrap_or_default()` is safe by inspection
- M4: Test that undoing a schema-category script delete restores the correct category
- M5: Test that purge strategy respects the configured limit; test that sync-enabled workspaces use the higher default
- M6: Test that `set_note_checked` writes a seconds-range timestamp
- M7: Test that importing a workspace preserves the original owner_pubkey

## Batch 3: Security Hardening — Input Validation

**GitHub label:** `Security: Input Validation`
**Issues addressed:** C2, C4, H1, H3, H4, M2

### C2 — Path traversal in `.swarmid` relay filename

**File:** `identity.rs:656`

`relay.filename` from deserialized `.swarmid` JSON is joined to a path unsanitized. A crafted filename with `../` can overwrite arbitrary files.

**Fix:** Sanitize with `Path::new(&relay.filename).file_name()` — accept only a bare filename with no path separators.

### C4 — Rhai engine has no operation limits

**File:** `scripting/engine.rs:17`

`Engine::new()` with no limits. A malicious or buggy user script with an infinite loop hangs the Tauri thread and locks the workspace Mutex forever.

**Fix:**
```rust
engine.set_max_operations(200_000);
engine.set_max_call_levels(64);
engine.set_max_string_size(1_000_000);
engine.set_max_array_size(100_000);
```

### H1 — SQLCipher key via string interpolation

**File:** `storage.rs:36-37, 61-62`

Single-quote escaping is fragile for unusual passwords.

**Fix:** Use hex-encoded literal form:
```rust
let hex_key = hex::encode(password.as_bytes());
conn.execute_batch(&format!("PRAGMA key = \"x'{hex_key}'\";\n"))?;
```

### H3 — `open_attachment` unsanitized temp filename

**File:** `commands/attachments.rs:202`

The filename (from DB, set at upload time) is joined to a temp path without sanitization.

**Fix:** Strip path separators with `Path::file_name()` before joining.

### H4 — `read_file_content` allows reading any file with matching suffix

**File:** `commands/workspace.rs:823`

Only checks file extension, no path confinement.

**Fix:** Canonicalize the resolved path and verify it falls within the workspace home directory or the themes directory. Reject paths that escape.

### M2 — Note titles logged at WARN in release builds

**File:** `commands/notes.rs:164-165`

`ghost_ops` target logs note titles at WARN level, included in production logs.

**Fix:** Remove the `ghost_ops` log lines entirely (debug instrumentation that was never cleaned up).

### Testing

- C2: Test with `../../../etc/passwd` filename, absolute path, symlink
- C4: Test with `loop {}` script — must return error, not hang
- H1: Verify encrypted DB opens correctly with the hex-encoded key
- H3: Test with `../../malicious` filename
- H4: Test with path traversal and symlink that escapes workspace dir
- M2: Verify WARN-level logs no longer contain note content

## Batch 4: Security Hardening — Sync & Crypto

**GitHub label:** `Security: Sync & Crypto`
**Issues addressed:** C1, H2, M1 + new sync_events table/UI

### C1 — Per-operation signatures not verified on delta ingest

**Files:** `swarm/delta.rs`, `swarm/sync.rs`, `workspace/sync.rs`

Operations are signed at creation but never verified on ingest. Any authenticated peer can inject forged operations.

**Fix:** Before `apply_incoming_operation`, call `op.verify(&sender_verifying_key)`. The method and key are already available. On failure, **reject the entire bundle** (not individual ops). Log the rejection to the new `sync_events` table.

### H2 — Relay account password not zeroized

**File:** `relay_account.rs:35`

`RelayAccount.password` is a plain `String` with `Clone + Debug` derives.

**Fix:** Wrap in `zeroize::Zeroizing<String>`. Remove `Debug` derive or implement custom `Debug` that redacts the password. Audit `.clone()` call sites.

### M1 — Attachment sidecar hashes excluded from bundle manifest

**File:** `swarm/delta.rs:107-111`

Sidecars can be silently stripped from bundles without invalidating the manifest signature.

**Fix:** Include `(filename, blake3_hash(ciphertext))` pairs for all `attachments/*.enc` files in the manifest hash input before signing. Verify on receive.

### New: `sync_events` table and UI

**Purpose:** Observable audit trail for sync failures so users can detect misbehaving peers.

**Schema:**
```sql
CREATE TABLE IF NOT EXISTS sync_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp INTEGER NOT NULL,
    peer_pubkey TEXT NOT NULL,
    event_type TEXT NOT NULL,
    detail TEXT
);
```

**Event types:** `bundle_rejected`, `signature_invalid`, `sidecar_mismatch`

**Tauri command:** `list_sync_events(window, limit, offset) -> Vec<SyncEvent>`

**UI:** Second tab in the existing Operations Log dialog. Shows timestamp, peer name (resolved from pubkey), event type, and detail. Minimal — a table with the data.

**i18n:** Add keys for tab label, column headers, and event type display names to all 7 locales.

### Testing

- C1: Test that a bundle with a tampered operation is rejected; test that a valid bundle is accepted
- H2: Verify `Debug` output does not contain password; verify password is zeroized (test with `zeroize` crate's test utilities if available)
- M1: Test that a bundle with a removed sidecar fails verification
- sync_events: Test that rejected bundles create entries; test `list_sync_events` query

## Batch 5: Tauri Command Fixes

**GitHub label:** `Tauri Command Fixes`
**Issues addressed:** C6, C7, M17, M18

### C6 + M17 — `lock_identity` window close race + stale paths

**File:** `commands/identity.rs:385-399`

`lock_identity` calls `win.close()` without inserting into `closing_windows`, so the close is intercepted by the frontend handler. State maps (`workspace_paths`, `workspace_identities`) are never cleaned up.

**Fix:** Insert label into `closing_windows`, then call `window.destroy()`. Explicitly remove entries from `workspace_paths`, `workspace_identities`, and `workspaces` maps. Match the `close_window` command pattern.

### C7 — Shared `pending_file_open` clobber

**File:** `commands/workspace.rs:111-149`

Both `.krillnotes` and `.swarm` file open handlers write to the same `Option<PathBuf>`, clobbering each other on rapid opens.

**Fix:** Split into `pending_krillnotes_open: Option<PathBuf>` and `pending_swarm_open: Option<PathBuf>` in `AppState`. Update `consume_pending_file_open` and `consume_pending_swarm_file` to read from their respective fields.

### M18 — `duplicate_workspace` concurrent open

**File:** `commands/workspace.rs:1144-1158`

Opens source DB without checking if it's already open in another window.

**Fix:** Check `find_window_for_path` at entry. If the workspace is currently open, return an error asking the user to close it first.

### Testing

Manual test checklist:
1. Lock identity with 2+ workspaces open — verify both windows close cleanly and no stale state remains
2. Rapid-open a `.krillnotes` file then a `.swarm` file — verify both are consumed
3. Attempt to duplicate a workspace that is currently open — verify error message
4. After lock/unlock cycle, verify the workspace can be reopened without issues

## Batch 7: Config & CI Hardening

**GitHub label:** `Config & CI Hardening`
**Issues addressed:** C3, H9, H10, M14, M15, M16 + zip cleanup

### C3 — Enable CSP

**File:** `tauri.conf.json:22`

**Fix:** Set:
```json
"security": {
  "csp": "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: asset: https://asset.localhost"
}
```

Add a comment in a nearby config file or README explaining how to extend the CSP for future embed/API features (add `frame-src` for embeds, `connect-src` for APIs).

**Implementation note:** Tauri v2 IPC uses a custom protocol (`ipc:` / `http://ipc.localhost`). Verify during implementation that the CSP does not break Tauri's `invoke()` calls — may need `connect-src 'self' ipc: http://ipc.localhost` explicitly. Test thoroughly before merging.

### H9 — Add CI workflow

**File:** `.github/workflows/ci.yml` (new)

Triggered on push and pull_request to `master`/`main`. Jobs:
- `cargo fmt --check`
- `cargo clippy -- -D warnings`
- `cargo test -p krillnotes-core`
- `cargo test -p krillnotes-rbac`
- `npx tsc --noEmit`
- `cargo audit` (via `actions-rs/audit-check` or equivalent)

### H10 — Bundle identifier

**File:** `tauri.conf.json:5`

Change `com.careck.krillnotes` to `com.2pisoftware.krillnotes`.

### M14 — Pin `tauri-action`

**File:** `.github/workflows/release.yml:53`

Pin `tauri-apps/tauri-action@v0` to a specific version tag or commit SHA.

### M15 — `rand` version

Audit `rand 0.9` compatibility across the crypto stack (`ed25519-dalek`, `x25519-dalek`, `chacha20poly1305`, `aes-gcm`). Upgrade if all deps support it. If not, document the blocker and revisit.

### M16 — `reqwest` blocking in core

Gate the `blocking` feature behind a `desktop` feature flag in `krillnotes-core/Cargo.toml`:
```toml
[features]
desktop = ["reqwest/blocking"]
relay = ["reqwest", "dep:reqwest"]
```

### Zip cleanup

Remove the bare `zip` dependency from `krillnotes-desktop/src-tauri/Cargo.toml` if it's not used directly (the features come through `krillnotes-core`).

### Testing

- CSP: Launch the app, verify it loads normally. Open DevTools console, verify no CSP violations for normal operation.
- CI: Push a test branch, verify all checks pass.
- Bundle ID: Build and verify the `.app` bundle uses the new identifier.

## Batch 8: macOS File Association Fix

**GitHub label:** `macOS File Association Fix`
**Issues addressed:** New investigation

### Scope

Investigate and fix the crash when opening `.krillnotes` files from macOS Finder.

### Investigation steps

1. Check `tauri.conf.json` file association config and generated `Info.plist` document type registration
2. Distinguish cold-start (app not running) vs. warm-start (app already running) — these are different Tauri v2 code paths
3. Capture Console.app crash logs to identify the actual panic point
4. Check if the file-open event fires before the app's main window is ready (race condition)
5. Review Tauri v2 GitHub issues for known macOS file association bugs

### Outcome

Fix if straightforward. If it's a Tauri v2 upstream bug, document findings and file an issue with the Tauri project.

### Testing

Manual:
1. With app closed: double-click a `.krillnotes` file in Finder — app should launch and open the workspace
2. With app running: double-click a `.krillnotes` file in Finder — app should open the workspace in a new window
3. With app closed: right-click → Open With → Krillnotes — same as (1)
